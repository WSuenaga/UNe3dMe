# State and Events
## 1．概要

本ページでは，UNe3dMe の UI 層における State 管理とイベント接続の流れを説明する．  
`demo.py` を中心として，UI コンポーネント，Gradio の `State`，バックエンド関数の関係を整理することを目的とする．

## 2．UI 層の役割

UI 層は主に `demo.py` に実装されており，以下を担当する．

- Gradio コンポーネントの定義
- タブや入力欄の構成
- ボタン・入力変更イベントの接続
- 言語切替に伴うラベル更新
- セッション単位の状態保持
- バックエンド関数の呼び出し

## 3．State 管理

### 3.1 State の目的

`gr.State` は，セッション中に共有したい値を保持するために用いる．  
これにより，前段の操作結果を後続の処理で利用できる．

### 3.2 主な State

本システムでは，たとえば以下の State を保持する．

- 一時ディレクトリ
- datasets ディレクトリ
- outputs ディレクトリ
- UI 言語状態
- 現在セットされている画像データセット
- 現在セットされている COLMAP データセット
- viewer スクリプト種別

### 3.3 State と実ファイルの違い

State は UI セッション内の参照値であり，ファイルそのものではない．  
実際の入力データや出力物は，`datasets/` や `outputs/` のディレクトリに保存される．  
したがって，State の更新とファイル生成は別の責務である．

## 4．言語切替の流れ

本システムは多言語 UI を想定しており，言語状態に応じてラベルや説明文を更新する．

### 4.1 主な関数
- `load_translations()`
- `update_ui()`

### 4.2 処理の流れ
1. 言語データを読み込む．
2. 現在の言語状態を参照する．
3. 各 UI コンポーネントのラベルや説明を更新する．

## 5．イベント接続の基本構造

Gradio のイベントは，UI 操作をバックエンド関数へ接続するために利用する．  
典型的な流れは以下のとおりである．

1. ユーザーが入力を行う．
2. ボタン押下または入力変更イベントが発火する．
3. `demo.py` で接続された関数が呼ばれる．
4. `local_backend.py` または `methods.py` が処理を実行する．
5. 処理終了後，戻り値が UI に反映される．

## 6．代表的なイベントフロー

### 6.1 画像からデータセットを作成する場合

- 入力
  - 画像ファイル群
  - データセット名

- 呼び出し先
  - `local_backend.copy_images()`

- 主な更新対象
  - 現在セットされている画像データセット State

### 6.2 動画からフレームを抽出する場合

- 入力
  - 動画ファイル
  - 抽出条件
  - 類似フレーム除去オプション

- 呼び出し先
  - `local_backend.extract_frames_with_filter()`

- 主な更新対象
  - 現在セットされている画像データセット State

### 6.3 COLMAP 前処理を行う場合

- 入力
  - 画像データセット
  - 再処理オプション

- 呼び出し先
  - `local_backend.run_colmap()`

- 主な更新対象
  - 実行ログ
  - 現在セットされている COLMAP データセット State

### 6.4 手法を実行する場合

- 入力
  - 対象データセット
  - 手法固有パラメータ
  - 実行モード

- 呼び出し先
  - `methods.recon_*()`

- 主な更新対象
  - 出力ディレクトリ
  - 実行時間
  - ステータス
  - 実行ログ

### 6.5 viewer を起動する場合

- 入力
  - viewer 種別
  - 対象 3D モデル
  - host / port

- 呼び出し先
  - `local_backend.viewer()`

- 主な更新対象
  - viewer URL

### 6.6 render / evaluation を行う場合

- 入力
  - 対象出力

- 呼び出し先
  - `methods.render_eval_*()` および `local_backend.evaluate_all_metrics()`

- 主な更新対象
  - 出力先
  - 実行時間
  - ステータス
  - 実行ログ
  - 評価結果
  - GT 画像とレンダリング画像

## 7．UI とバックエンドの責務分離

責務分離の方針は以下のとおりである．

### 7.1 `demo.py`
- UI コンポーネントを定義する．
- State を保持する．
- イベントを接続する．
- 画面更新を行う．

### 7.2 `local_backend.py`
- データ準備，前処理，共通処理を行う．
- ファイル操作，viewer 起動，評価を行う．

### 7.3 `methods.py`
- 手法ごとのコマンド組み立てと実行を行う．
- 実行結果を UI が扱いやすい形式で返す．

## 8．戻り値設計

UI に接続される関数は，Gradio コンポーネント更新に必要な形式で値を返す必要がある．  
本システムでは，たとえば以下のような戻り値を返す．

- 出力ディレクトリ
- 実行時間
- ステータス
- ログ文字列
- update オブジェクト
- 更新後の State 値

戻り値の順序と意味は，イベント接続側と関数定義側で一致させる必要がある．

## 9．実装上の注意点

### 9.1 戻り値の個数を一致させる
Gradio のイベント出力数と，関数の戻り値数は一致させる必要がある．

### 9.2 言語更新と値更新を分離する
ラベル更新処理と言語に依存しない値更新処理は，可能な限り分離した方が保守しやすい．
